## 시놉시스 
1인부터 4인 멀티플레이 까지 지원하는 쿼터뷰 액션 슈팅 게임입니다. 
빙하의 생성과 파괴를 주요 게임 시스템으로 활용합니다.
펭귄인 플레이어는 살아남기 위해선 빙하로부터 떨어지지 않아야합니다. 빙하를 녹이고 펭귄을 떨어트리려 몰려오는 적들을 물리치고 바다를 얼려 살아남으세요! 

<br>

## 주요 기능
###### 멀티플레이
1인부터 4인까지 함께 플레이 가능합니다.
> Steam Advanced Session 을 이용한 listen Server 방식입니다.

###### 플레이어 이동 및 공격 
키보드 방향키를 통해 상하좌우 이동이 가능하며 마우스로 방향을 조준하고 클릭하면 총을 발사합니다. 발사된 총알은 적이나 몬스터에 닿으면 범위형으로 데미지를 가하고 주변에 일정 영역만큼 빙하를 생성합니다.

###### 빙하 지형 시스템
삼각형 그리드로 빙하가 생성되고 파괴됩니다. 빙하의 크기는 체력에 비례합니다.

###### 몬스터
오징어: 플레이어 방향으로 헤엄치며 다가옵니다. 몸에 닿는 빙하에 데미지를 주며 파괴합니다. <br>
꽃게: 빙하 위로 올라와 돌진을 통해 플레이어를 빙하위에서 밀어냅니다.

###### 웨이브
Wave가 준비되어 있고 각 Wave 별로 몬스터가 소환됩니다. 모든 적을 잡아야 다음 Wave로 넘어갑니다. 한 라운드가 진행되는 시간에 비례해서 빙하가 점차 빠르게 녹습니다. 다음 라운드로 넘어가면 녹는 속도가 0으로 초기화됩니다. 

###### UI
메인 화면, 대기화면, 인게임, ESC 설정, 게임 패배 / 게임 승리 의 UI들로 게임 플레이를 보조합니다.

## 리팩토링 일화 
### 1. 메모리 최적화 및 가비지 컬렉션(GC) 관리

#### **1.1. 비헤이비어 트리(BT) 메모리 최적화 및 인스턴싱 구조 리팩토링**

- **문제:** 매 태스크 실행 시 **객체를 동적 할당해 CPU 오버헤드가 발생 및 GC 발생**
- **해결:** **노드 인스턴싱 제거**, 노드 객체는 1개만 유지, 상태 데이터는 가벼운 구조체 **NodeMemory에** 재활용
- **측정 환경 :** 꽃게 몬스터 50마리(20초), 오징어 몬스터 10마리(30초)
- **최적화 결과:** GC 발생량을 최대 **50.6%까지** 감소, 주요 연산 시간 감소
    
| **대상** | **메인 메모리** | **실시간 메모리 할당** | **가비지 컬렉션(GC)** | **주요 총 연산 시간** |
| :--- | :--- | :--- | :--- | :--- |
| **꽃게** | 0.24% ▼ | 36.5% ▼ | 50.6% ▼ | BTTask_PushMove 23.08% ▼<br>BTTask_IsSwimming 18.87% ▼ |
| **오징어** | 1.2% ▼ | 16.2% ▼ | 2.15% ▼ | BTTask_SquidMove 31.56% ▼ |

#### 1.2. 몬스터 및 총알 오브젝트 풀링(Object Pooling) 시스템 도입

- **문제**: 몬스터, 총알 스폰시 **SpawnActor/Destroy를 반복 호출해** 병목 발생, **기존 풀 관리자의 확장성 부족**
- **해결**: **인터페이스 기반의 범용 오브젝트 풀 매니저**를 사용해 객체를 재사용 하는 구조로 변경
- **측정 환경:** 몬스터 스폰(1R 각 2마리 → 2R 각 4마리 → 3R 각 8마리 → 4R 각 16마리), 총알 20회 발사
- **최적화 결과**: 몬스터 소환 연산 시간 67.6% 감소, 총알 소환 연산 시간 81.5% 감소
    
| **대상** | **메인 메모리** | **가비지 컬렉션(GC)** | **주요 연산 시간** |
| :--- | :--- | :--- | :--- |
| **몬스터 스폰** | 1.3% ▲ | 5.1% ▼ | SpawnEnemy 67.6% ▼ |
| **총알 스폰** | " | " | Shoot 81.5% ▼<br>BP_Bullet 71.1% ▼ |

#### 1.3. 나이아가라 이펙트 컴포넌트 재사용

- **문제**: 사격 시마다 이펙트 시스템 **동적 생성/부착해 GC 오버헤드** 발생

### 2. CPU 연산 및 물리 엔진 최적화

#### 2.1. 옵저버 패턴 기반 센싱 로직 최적화

- **문제**: 매 틱(Tick)마다 **월드 내 모든 액터를 탐색(O(N))해** 불필요한 CPU 연산과 가상 함수 호출 비용이 누적
- **해결**: **옵저버 패턴**을 도입해 Tick 검사 제거. **델리게이트 기반 이벤트 구조**로 리팩토링해 탐색 비용을 **O(1)로** 단축
- **측정 환경 :** 플레이어 추적(꽃게 20마리), 플레이어 상태 확인(20초 뒤 펭귄 자살), 빙하 개수 전수 조사(오징어 20마리)
- **최적화 결과 :** 주요 연산시간 감소

| **플레이어 추적** | 0.2% ▼ | 2.7% ▼ | BTTask_ChasePlayer 44.8% ▼ |

#### 2.2. 물리 판정 최적화 및 필터링

#### **2.3. 컴포넌트 생성 로직 비동기 공간 쿼리로 교체**

### 3. 아키텍처 확장성 및 네트워크 안정성

#### 3.1. 몬스터 공통 로직 추상화 리팩토링

몬스터 클래스 **공통 로직을 Base 클래스로 추상화**해 코드 중복을 80% 이상 제거하고 확장성 확보

#### 3.2. 러버밴딩 현상 및 멀티플레이 동기화 오류 해결

- **문제** : 클라이언트 환경에서 **몬스터 사망시 콜리전 해제 동기화 누락돼 플레이어 이동시 러버밴딩 발생**
- **해결 및 결과** : **서버와 클라이언트 간의 상태 동기화(GetLifetimeReplicatedProps)로** 해결

#### 3.3. 델리게이트 중복 할당 성능 저하 해결

- **문제** : 몬스터 첫 생성시, 플레이어 첫 사망시 **순간적인 프레임 드랍** 발생
- **해결** **및 결과** : 이벤트가 한번만 바인딩 되게 수정, **AddUniqueDynamic으로** 중복 할당 방지
#### 3.4. 몬스터 AI 구조 개선 및 역할 분리, 이벤트 다중 바인딩 방지

- **문제:** BTTask에서 충돌 물리, 상태 관리, 타이머 기능을 제어해 노드 호출시마다 **불필요한 연산 및 메모리 오버헤드** 발생
- **해결:** 몬스터 액터에 물리, 상태, 타이머 제어 역할을 Pawn클래스로 이동시키고 **BTTask는 AI 제어 및 이동 로직만 수행**
- **측정 환경**: 10마리 오징어 소환, 30초 뒤 플레이 종료
- **최적화 결과 :** GC 68% 감소 및 주요 연산시간 감소

| **메인 메모리** | **실시간 메모리 할당** | **가비지 컬렉션(GC)** | **주요 연산 시간** |
| :--- | :--- | :--- | :--- |
| 0.63% ▲ | 21.1% ▼ | 68% ▼ | BP_Squid_C 26.7% ▼<br>BTTask_SquidMove 41.4% ▼ |

#### 3.5 클라이언트 예측 기반 사격 이벤트 최적화 및 동기화

- **문제**: 사격 입력 → 서버 보고→ 이펙트, 투사체 생성 순서로 처리돼 **응답 지연** 발생
- **해결 방법**: **클라이언트 예측 방법을** 써 로컬에서 이펙트, 가짜 총알 스폰, 서버 RPC, **Unreliable Multicast로** 동기화
- **측정 환경**: 네트워크 핑 100ms 환경에서 클라이언트에서 총 1번 발사

#### 3.6. NetUpdateFrequency, NetDormancy를 통한 네트워크 최적화

- **문제**: 빙하, 몬스터에서 **빈번한 네트워크 동기화로** 시간이 지날수록 클라이언트 측 프레임 저하 발생
- **해결 방법**: NetUpdateFrequency로 **갱신주기 조절** 및 SetNetDormancy() 로 **네트워크 수면상태** 적용
- **측정 환경**: 4라운드까지 진행 후 게임 종료
- **최적화 결과** :  서버 네트워크 갱신 부하 55% 감소, 클라이언트 수신 패킷 처리 36% 감소

| :--- | :--- | :--- |
| 5.9% ▼ | 33.4% ▼ | GameNetDriver 55% ▼ |
